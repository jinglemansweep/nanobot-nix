# planner-create-plan

Generate a high-level implementation plan from a prompt file. This is the first stage of a two-step workflow — the plan produced here feeds into `planner-create-tasklist` which breaks it down into granular tasks.

## Usage

```
/planner-create-plan <plan-dir>
```

**Argument:**
- `<plan-dir>` — Path to the plan directory (e.g. `.agent/plans/01-initial`). This directory must already contain a `PROMPT.md` file.

**Directory convention:**

```
<plan-dir>/
├── PROMPT.md       ← input  (must exist)
├── RESEARCH.md     ← output (generated by this skill)
├── PLAN.md         ← output (generated by this skill)
├── CONCERNS.md     ← output (generated by this skill)
└── TASKS.md        ← output (generated later by planner-create-tasklist)
```

## Instructions

### Step 1: Read the Prompt

Read `<plan-dir>/PROMPT.md` in full. If the file or directory does not exist, stop and tell the user.

### Step 2: Research

Scan the prompt for any external references — URLs, GitHub repos, library/tool names, APIs, specifications, or third-party projects. For each reference found:

1. **Fetch and review** the link or search for the project to determine:
   - Current status (actively maintained? latest release/version?)
   - Compatibility with the project's stack and constraints
   - Known limitations, breaking changes, or deprecations
   - Installation/integration requirements
   - Licence compatibility
2. **Record findings** in `<plan-dir>/RESEARCH.md`.

Structure `RESEARCH.md` as follows:

```markdown
# Research

> Source: `<plan-dir>/PROMPT.md`

## <Reference/Project/Link Name>
- **URL:** <link>
- **Status:** <actively maintained / archived / deprecated / unknown>
- **Latest Version:** <version or N/A>
- **Compatibility:** <assessment against project stack>
- **Key Findings:** <bullet points of relevant details>
- **Concerns:** <any issues, limitations, or risks discovered>

## ...
```

If the prompt contains no external references, still create `RESEARCH.md` with a note that no external references were found.

### Step 3: Analyze

Analyze the prompt (and research findings) thoroughly. Identify:
- The project's goals and scope
- Key decisions and constraints already made
- Components, systems, or modules that need to be built
- Dependencies and ordering between components
- Any ambiguities, gaps, risks, or potential future problems

### Step 4: Generate the Plan

Generate `<plan-dir>/PLAN.md` structured as follows:

```markdown
# Implementation Plan

> Source: `<plan-dir>/PROMPT.md`

## Overview
<!-- 2-3 sentence summary of what this plan delivers -->

## Architecture & Approach
<!-- Describe the overall technical approach, key design choices,
     and how the components fit together. Reference decisions from
     the prompt and findings from research. No code examples unless
     absolutely essential to convey the approach. -->

## Components
<!-- One subsection per major component/module/system. Describe
     WHAT it does, WHY it exists, its responsibilities, and how
     it relates to other components. Do NOT include task lists,
     checklists, or step-by-step instructions — those belong in
     the tasklist stage. -->

### <Component Name>
**Purpose:** <what this component is responsible for>
**Inputs:** <what it receives or depends on>
**Outputs:** <what it produces or exposes>
**Notes:** <key design considerations, edge cases, constraints>

## File Manifest
<!-- Table of every file to be created or modified -->
| File | Action | Purpose |
|------|--------|---------|
| `path/to/file` | Create / Modify | Brief description |
```

### Step 5: Generate Concerns

Collect all open questions, unresolved issues, potential blockers, risks, and future concerns identified during analysis and research. Write to `<plan-dir>/CONCERNS.md`.

Structure `CONCERNS.md` as follows:

```markdown
# Concerns

> Source: `<plan-dir>/PROMPT.md`

## Open Questions
<!-- Ambiguities or decisions not yet resolved in the prompt -->
- ...

## Potential Blockers
<!-- Issues that could prevent or significantly delay implementation -->
- ...

## Risks
<!-- Things that might go wrong or cause problems during or after implementation -->
- ...

## Future Considerations
<!-- Not blockers now, but likely to become relevant later -->
- ...
```

### Step 6: Report

Report a brief summary to the user confirming what was written to `<plan-dir>/`:
- Number of components and files identified in the plan
- Number of references researched
- Number of concerns raised

## Guidelines

- **This is a PLAN, not a tasklist.** Focus on architecture, components, responsibilities, and relationships. Do not produce task lists, checklists, phases, milestones, stages, or step-by-step instructions — that is the job of `planner-create-tasklist`.
- Do not include code examples unless they are essential to explain an approach that cannot be conveyed in prose.
- Reference specific files and paths from the prompt wherever possible.
- If the prompt contains explicit decisions (e.g. a decisions table), respect them — do not second-guess or re-open settled choices.
- Research should be thorough but focused — only investigate references that are relevant to implementation decisions. Don't research generic or well-known tools (e.g. `git`, `curl`) unless the prompt raises a specific compatibility question about them.
- Concerns should be genuine and actionable, not padded. If there are no concerns in a category, say so briefly rather than inventing issues.
- The plan should give a complete picture of *what* is being built and *why*, so that the subsequent tasklist stage can derive *how* without re-reading the original prompt.
